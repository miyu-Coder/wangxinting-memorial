# 技术亮点与系统设计框架

## 第一部分：技术亮点

### 一、代码质量亮点

#### 1.1 模块化架构设计

**亮点名称**：前端功能模块化 + 后端路由分层

**代码体现**：
- 前端按功能拆分为 20+ 独立 JS 模块（checkin.js、flowers.js、messages.js、achievement.js 等），每个模块职责单一
- 后端预留 `server/routes/` 目录分层（admin.js、checkin.js、flower.js、quiz.js、messages.js、stats.js）
- 公共逻辑抽离至 common.js（昵称管理、庆祝动画、留言墙渲染）和 api.js（请求封装）

**论文描述**：系统采用模块化架构设计，前端按业务功能拆分为独立模块，每个模块封装单一职责的交互逻辑；后端按资源类型组织路由层，公共逻辑统一抽离，实现了高内聚低耦合的代码组织方式，便于功能扩展和维护。

#### 1.2 数据库 Promise 化封装

**亮点名称**：回调转 Promise 的数据库操作层

**代码体现**：
```javascript
db.runAsync = (sql, params = []) => {
  return new Promise((resolve, reject) => {
    db.run(sql, params, function (err) {
      if (err) reject(err);
      else resolve(this);
    });
  });
};
db.getAsync / db.allAsync 同理
```

**论文描述**：针对 SQLite3 驱动的回调式 API，系统封装了 Promise 化的数据库操作层（runAsync/getAsync/allAsync），使所有数据库操作支持 async/await 语法，消除了回调地狱问题，提升了代码可读性和异常处理能力。

#### 1.3 软删除机制

**亮点名称**：留言审核的软删除策略

**代码体现**：
- messages 表 status 字段：0=待审核、1=通过、2=拒绝
- 拒绝操作 `UPDATE messages SET status = 2` 而非 DELETE
- 前端查询 `WHERE status = 1` 只展示通过的内容

**论文描述**：系统采用软删除机制管理用户留言，通过状态字段（0/1/2）而非物理删除来标记数据生命周期。这种设计保留了完整的数据轨迹，支持误操作恢复，同时为后续的内容审核分析和数据挖掘提供了数据基础。

#### 1.4 防重复提交机制

**亮点名称**：多层级去重策略

**代码体现**：
- 数据库层：`UNIQUE(user_identifier, exhibit_id)` 约束防止重复打卡/献花
- 业务层：答题记录查询 `SELECT id FROM quiz_records WHERE nickname = ? AND exhibit_id = ?` 去重
- 业务层：页面访问 10 分钟去重 `WHERE visit_time > datetime('now', '-10 minutes')`
- 业务层：纪念品预约按 order_type 区分去重条件

**论文描述**：系统实现了多层级的数据去重策略。数据库层通过 UNIQUE 约束保证打卡和献花的唯一性；业务层通过查询-插入模式和时效窗口机制，防止答题记录重复提交和页面访问的短时间重复计数，确保数据的准确性和统计的有效性。

### 二、用户体验亮点

#### 2.1 适老化设计

**亮点名称**：全局字体调节与记忆功能

**代码体现**：
- font-size.js 提供全局字体大小调节控件
- 调节结果通过 localStorage 持久化
- 页面加载时自动恢复用户偏好设置

**论文描述**：针对红色教育基地参观者中老年群体较多的特点，系统实现了适老化设计。提供全局字体大小调节功能，用户设置通过 localStorage 持久化存储，再次访问时自动恢复偏好设置，降低了老年用户的使用门槛。

#### 2.2 无登录用户追踪

**亮点名称**：基于 IP+UA 指纹的无感用户标识

**代码体现**：
```javascript
app.use((req, res, next) => {
  const ip = req.ip || req.connection.remoteAddress || '127.0.0.1';
  const ua = req.get('User-Agent') || '';
  req.userIdentifier = crypto.createHash('md5').update(ip + ua).digest('hex').substring(0, 16);
  next();
});
```

**论文描述**：系统设计了无需用户注册登录的身份标识方案。通过将访问者 IP 地址与浏览器 User-Agent 字符串拼接后进行 MD5 哈希，生成 16 位用户指纹作为唯一标识。该方案在保护用户隐私（不采集明文信息）的前提下，实现了跨页面的用户行为追踪，用于打卡去重和访问统计。

#### 2.3 庆祝动画反馈

**亮点名称**：Canvas 全屏粒子动画

**代码体现**：
- common.js 中 drawStar/drawFlower 函数实现全屏 Canvas 粒子动画
- 打卡成功触发五角星飘落动画，献花成功触发花瓣飘落动画
- 粒子参数（数量、速度、光晕大小）统一标准化

**论文描述**：系统采用 HTML5 Canvas 技术实现了全屏粒子动画效果。当用户完成打卡或献花操作时，分别触发五角星和花瓣的飘落动画，通过视觉反馈增强用户的成就感和参与感。动画采用 requestAnimationFrame 驱动，保证流畅性的同时控制性能开销。

#### 2.4 留言墙黑板粉笔风格

**亮点名称**：多维度随机化的粉笔留言渲染

**代码体现**：
- 9 种颜色（含深红、深蓝对比色）
- 12 级字号（14px-30px），1-2 条强制大号
- 11 种字体（含行书、草书等书法字体）
- 30% 概率倾斜、±5° 随机旋转
- 短留言（≤15字）并排显示，长留言独占一行

**论文描述**：留言墙采用黑板粉笔的视觉隐喻，通过多维度随机化渲染（字体、字号、颜色、倾斜角度、旋转角度）模拟真实粉笔书写的自然差异。短留言采用 inline-block 并排布局提升空间利用率，长留言独占一行保证可读性，实现了美观与信息密度的平衡。

### 三、数据设计亮点

#### 3.1 展点转化率分析

**亮点名称**：浏览→打卡的转化率计算

**代码体现**：
```javascript
const conversionRate = viewCount > 0 
  ? Math.round(checkinCount * 1000 / viewCount) / 10 
  : 0;
```
- page_views 记录浏览，visits 记录打卡
- 按 exhibit_id 关联计算转化率

**论文描述**：系统设计了展点转化率分析模型，通过关联页面浏览记录（page_views）与打卡记录（visits），计算各展点从浏览到打卡的转化率。该指标帮助管理者识别哪些展点的互动设计更有效，为展陈优化提供数据支撑。

#### 3.2 排行榜综合评分算法

**亮点名称**：加权综合排名算法

**代码体现**：
```
排名分 = 得分率 × 50% + 完成度 × 30% + 速度分 × 20%
排序：先按 completed_exhibits 降序，再按 ranking_score 降序
```
- SQL 子查询先按 nickname+exhibit_id 取 MAX(score) 去重
- 确保完成全部 4 展点的用户排在前列

**论文描述**：排行榜采用加权综合评分算法，将得分率（50%）、完成度（30%）和速度分（20%）三个维度加权求和。排序时优先按完成展点数降序排列，确保完成全部答题的用户始终排在部分完成的用户之前，避免单一维度（如满分但只完成1个展点）导致的排名失真。数据查询层通过子查询对同一用户同一展点取最高分，消除重复答题对总分的干扰。

#### 3.3 演示数据的加权分布

**亮点名称**：模拟真实访问模式的演示数据生成

**代码体现**：
- 展点访问加权分布（陈列馆30%、故居25%、广场25%、装备20%）
- 答题分数加权分布（满分30%、3分35%、2分25%、1分10%）
- 周末访问量自动增加 30%-50%
- Fisher-Yates 洗牌算法确保昵称/留言均匀分布
- 同一用户同一展点只生成一条答题记录

**论文描述**：系统的演示数据生成器采用加权随机分布模拟真实访问模式。展点访问量按热度分配权重，答题分数按正态分布近似分配，周末访问量自动上浮。通过 Fisher-Yates 洗牌算法保证昵称和留言的均匀分布，避免数据重复集中，使演示数据更贴近真实场景。

#### 3.4 会话级 PV 去重

**亮点名称**：10 分钟窗口的页面访问去重

**代码体现**：
```sql
SELECT id FROM page_views 
WHERE page = ? AND session_id = ? 
AND visit_time > datetime('now', '-10 minutes')
```

**论文描述**：页面访问统计采用会话级去重策略，在 10 分钟时间窗口内，同一会话对同一页面的重复访问不重复计数。该机制有效过滤了用户刷新页面、浏览器自动重载等场景产生的无效 PV，提升了访问统计的准确性。

### 四、安全设计亮点

#### 4.1 环境变量管理

**亮点名称**：敏感配置的环境变量隔离

**代码体现**：
- dotenv 管理环境变量
- ADMIN_PASSWORD 通过 process.env.ADMIN_PASSWORD 读取
- .env 文件加入 .gitignore 不提交版本控制
- 提供 .env.example 作为配置模板

**论文描述**：系统采用环境变量管理敏感配置，管理员密码等关键信息通过 dotenv 从环境变量读取，不硬编码在源码中。.env 配置文件排除在版本控制之外，同时提供 .env.example 模板文件指导部署配置，实现了安全性与便利性的平衡。

#### 4.2 接口输入校验

**亮点名称**：全接口参数校验与类型检查

**代码体现**：
- 展点 ID 范围校验：`![1, 2, 3, 4].includes(Number(exhibitId))`
- 手机号正则校验：`/^1\d{10}$/.test(phone)`
- 昵称/内容长度校验：`Array.from(nick).length > 20`
- Unicode 正确计数：使用 `Array.from()` 而非 `.length` 计算中文字符数

**论文描述**：系统对所有 API 接口实施严格的输入校验。参数类型、取值范围、格式规范均在前端和后端双重验证。特别地，针对中文文本的长度校验采用 `Array.from()` 方法正确计数 Unicode 字符，避免了传统 `.length` 属性对 emoji 和多字节字符的误判。

#### 4.3 安全中间件

**亮点名称**：Helmet 安全头注入

**代码体现**：
- helmet 中间件自动注入安全响应头（XSS 防护、CSP、HSTS 等）

**论文描述**：系统集成 Helmet 安全中间件，自动为 HTTP 响应注入安全头信息，包括 XSS 防护、内容安全策略（CSP）、HTTP 严格传输安全（HSTS）等，有效防御常见的 Web 安全威胁。

## 第二部分：系统设计章节框架

### 第三章 系统设计

#### 3.1 系统架构设计

**3.1.1 总体架构**

系统采用前后端一体的 B/S 架构，Express 同时承担静态资源托管和 API 服务角色。前端页面通过 fetch 调用 RESTful API 获取数据，服务端从 SQLite 数据库查询后返回 JSON 格式响应。

```
┌──────────────────────────────────────────────┐
│                  浏览器端                      │
│  ┌─────────┐ ┌──────────┐ ┌───────────────┐ │
│  │ 首页模块 │ │ 展点详情  │ │ 成就/证书/排行 │ │
│  └────┬────┘ └────┬─────┘ └──────┬────────┘ │
│       └──────────┬─┘              │           │
│            ┌─────┴─────┐          │           │
│            │  api.js   │          │           │
│            └─────┬─────┘          │           │
└──────────────────┼────────────────┘           │
                   │ HTTP / RESTful API          │
┌──────────────────┼────────────────────────────┐
│            ┌─────┴─────┐                      │
│            │ Express 5 │                      │
│            │ server.js │                      │
│            └─────┬─────┘                      │
│       ┌──────────┼──────────┐                 │
│  ┌────┴────┐ ┌───┴───┐ ┌───┴────┐           │
│  │ API路由  │ │静态托管│ │安全中间件│           │
│  └────┬────┘ └───────┘ └────────┘           │
│       │                                       │
│  ┌────┴────┐                                 │
│  │ SQLite  │                                 │
│  │ data.db │                                 │
│  └─────────┘                                 │
└──────────────────────────────────────────────┘
```

**3.1.2 前后端分离策略**

前端采用多页面应用（MPA）架构，每个功能页面独立 HTML 文件，通过公共 JS 模块共享逻辑。后端提供纯数据 API，不参与页面渲染，实现了前后端职责的清晰分离。

**3.1.3 数据驱动渲染**

前端页面通过 fetch 获取 JSON 数据后动态渲染 DOM，展点内容存储在 data.json 中，管理后台修改后前端即时生效，无需重新部署。

#### 3.2 数据库设计

**3.2.1 表结构设计**

系统共设计 7 张数据表，覆盖用户交互（visits、flowers、quiz_records、messages）、访问统计（page_views）、业务管理（souvenir_orders）和系统运维（admin_logs）四个领域。

**3.2.2 约束策略**

| 表 | 约束类型 | 实现方式 | 目的 |
|---|---|---|---|
| visits | 复合唯一约束 | UNIQUE(user_identifier, exhibit_id) | 防止重复打卡 |
| flowers | 复合唯一约束 | UNIQUE(user_identifier, exhibit_id) | 防止重复献花 |
| quiz_records | 业务层去重 | 先查后插 | 防止重复答题 |
| page_views | 时效窗口去重 | 10分钟内不重复计数 | 过滤无效PV |
| messages | 软删除 | status字段标记 | 保留数据轨迹 |
| souvenir_orders | 业务层去重 | 按类型+展点/排名查重 | 防止重复预约 |

**3.2.3 数据库 Promise 封装**

对 SQLite3 回调式 API 进行 Promise 化封装，支持 async/await 语法，统一异常处理模式。

#### 3.3 用户追踪方案

**3.3.1 无登录设计**

系统不要求用户注册登录，降低使用门槛，特别适合红色教育基地的即来即用场景。

**3.3.2 用户指纹生成**

```
IP地址 + User-Agent → MD5哈希 → 取前16位 → user_identifier
```

优势：无需用户配合即可区分不同设备；隐私保护（不存储明文 IP/UA）；跨页面一致性。

局限：同一设备不同浏览器生成不同指纹；NAT 网络下多设备可能同 IP。

**3.3.3 会话追踪**

通过前端生成的 session_id（localStorage 持久化）关联页面访问记录，用于 UV 统计和用户路径分析。

#### 3.4 核心功能实现

**3.4.1 打卡功能**

流程：用户进入展点页 → 前端查询打卡状态 → 未打卡则显示打卡按钮 → 点击打卡 → API 写入 visits 表 → 触发庆祝动画 → 按钮变为"已打卡"状态。

去重：数据库 UNIQUE 约束 + 前端状态缓存双保险。

**3.4.2 献花功能**

流程与打卡类似，独立去重。额外提供近期献花名单查询，增强社交氛围。

**3.4.3 答题功能**

- 每展点 4 道选择题，数据存储在 data.json
- 前端即时评分，记录答题用时（time_cost）
- 满分用户可预约纪念品
- 排行榜按子查询去重后聚合（每展点取最高分）

**3.4.4 留言功能**

- 提交留言进入待审核状态（status=0）
- 管理员审核通过（status=1）后前端可见
- 拒绝采用软删除（status=2），保留数据
- 前端黑板粉笔风格渲染，多维度随机化

**3.4.5 排行榜**

综合评分算法：得分率×50% + 完成度×30% + 速度分×20%
排序策略：完成展点数优先，同完成数按评分排序

**3.4.6 纪念品预约**

两种预约类型：
- 展点预约（order_type=exhibit）：单展点满分可预约对应奖品
- 排行榜预约（order_type=ranking）：按排名自动分配奖品名称
- 四展点全满分解锁将军纪念礼盒

#### 3.5 后台管理系统

**3.5.1 数据看板**

- ECharts 可视化：7日趋势折线图、时段分布柱状图
- 核心指标：PV/UV、转化率、献花数、热门展点
- 展点转化率 = 打卡数 / 浏览数 × 100%

**3.5.2 内容管理**

- 展点内容编辑（标题、简介、正文、音频、视频）
- 答题题目管理
- iframe 实时预览
- CSV 批量导入导出

**3.5.3 审核机制**

- 留言审核：通过/拒绝/删除
- 软删除保留数据轨迹
- 操作日志全程记录

**3.5.4 智能分析**

- 答题时长统计（平均用时、最快用时、各展点用时分布）
- 答题高级统计（完成率、满分率、独立用户数）
- 展点转化率对比分析

#### 3.6 部署方案

**3.6.1 本地开发环境**

```bash
npm install && npm run dev
```

nodemon 监听文件变化自动重启，SQLite 零配置即用。

**3.6.2 生产部署**

- Node.js 单进程部署，端口通过环境变量配置
- SQLite 文件数据库，无需独立数据库服务
- 静态资源由 Express 直接托管
- 适合中小规模红色教育基地的低成本部署需求

**3.6.3 云端部署**

项目包含 vercel.json 和 netlify.toml 配置文件，支持 Vercel/Netlify 等云平台一键部署（需配合 Serverless SQLite 方案）。
